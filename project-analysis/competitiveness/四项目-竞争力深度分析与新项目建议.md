### 2.1 notification-platform

**定位**: 多渠道消息通知平台（短信/邮件/站内信/推送），支持模板管理、批量发送、回调追踪

#### 🔴 严重问题（面试致命级）

**问题 1: 服务层零单元测试**

整个 `internal/service/` 目录没有一个 `_test.go` 文件。现有测试全部是 `integration/` 目录下的集成测试，依赖真实 MySQL 和 Redis。

**问题 2: 10+ 处静默吞错**


**影响**: 在分布式系统中，静默吞错是最危险的模式之一。JSON 序列化失败意味着数据丢失，但没有任何告警或日志。

**问题 3: BatchSubmitForProviderReview 吞掉所有错误**

### 2.2 distributed_task_platform

**定位**: 分布式任务调度平台，支持普通/分片/DAG 任务、延时执行、失败补偿  
**亮点**: 架构设计最为完整，SchedulerV1 有 WaitGroup 优雅退出，DAG 编排有深度

#### 🔴 严重问题

**问题 1: 核心模块零测试**

```
// 测试覆盖统计：
// 87 个源文件 vs 11 个测试文件
// 以下核心模块 0 测试：
//   - scheduler/ (调度器)
//   - runner/ (执行器)
//   - compensator/ (补偿器)
//   - invoker/ (调用器)
//   - loadchecker/ (负载检测)
```

**影响**: 这五个模块是整个平台的核心引擎。零测试意味着任何重构都是"盲改"，面试官会质疑这个项目在生产环境的可靠性。

**问题 2: SchedulerV2 优雅退出有缺陷**

```go
// internal/service/scheduler/schedulerv2.go:110-119
func (s *SchedulerV2) Start(ctx context.Context) error {
    go s.scheduleLoop(ctx)    // 启动 goroutine
    go s.monitorLoop(ctx)     // 启动 goroutine
    // 没有 WaitGroup！无法追踪这些 goroutine
    return nil
}

// 对比 V1 的正确做法：
// internal/service/scheduler/scheduler.go:117
// s.wg.Add(2)  // V1 用 WaitGroup 追踪

// schedulerv2.go:222-226
func (s *SchedulerV2) GracefulStop() {
    s.cancel()  // 只发了取消信号
    // 但没有 wg.Wait()！不等 goroutine 结束就返回了
    // 可能导致：正在执行的任务被强制中断、数据库状态不一致
}
```

**影响**: 这是一个架构级 bug。面试官问"你的调度器怎么做优雅退出？"，你说用 context cancel + WaitGroup，但实际代码 V2 版本根本没有 WaitGroup。如果面试官看过代码会很尴尬。

**问题 3: plan.go 变量名 bug**

```go
// internal/service/task/plan.go:63-65
_, eerr := s.executionRepo.UpdateStatus(ctx, executionID, status)
if eerr != nil {
    return err  // BUG: 应该是 return eerr，这里返回的是外层的 err（可能为 nil）
}
```

**影响**: 这个 bug 会导致 UpdateStatus 失败时，函数返回 nil 而不是错误。调用方以为更新成功了，但实际上执行计划的状态没有被正确更新。

**问题 4: 生产代码中的 panic**

```go
// internal/repository/dao/sharding_task_execution.go:494-496
func (d *ShardingTaskExecutionDAO) SomeMethod() {
    panic("implement me")  // 分片任务的核心 DAO 方法没实现
}
```

**问题 5: MQ Consumer 启动失败直接 panic**

```go
// internal/event/reportevt/report_event_consumer.go:50-52
if err != nil {
    panic(err)  // 消费者启动失败 → 整个进程 crash
    // 应该返回 error 让上层决定是否重试
}
```

#### 🟡 中等问题

**问题 6: RetryConfig 空指针风险**

```go
// internal/service/task/execution_service.go:370
retryStrategy, _ := retry.NewRetry(execution.Task.RetryConfig.ToRetryComponentConfig())
// 如果 Task.RetryConfig 为 nil → panic: nil pointer dereference
// 而且 error 也被忽略了
```

**问题 7: Runner goroutine 泄漏风险**

```go
// internal/service/runner/normal_task_runner.go:123-138
go func() {
    // 执行任务逻辑
    // 但这个 goroutine 没有被任何 WaitGroup 或 channel 追踪
    // 如果主进程退出，这个 goroutine 可能来不及完成
}()
```

### 2.3 ai-dataplatform-ops

**定位**: AI 驱动的数据平台智能运维 Agent，结合 LangGraph 工作流 + MCP 工具 + RAG 检索  
**亮点**: 概念超前（AI Agent + 大数据运维是当前最热方向之一），架构设计清晰

#### 🔴 严重问题（这个项目的问题性质不同——不是 bug，是"还没做完"）

**问题 1: 全部 20+ Go MCP 工具返回硬编码 Mock 数据**

```go
// go/internal/tools/hdfs/tools.go:30-86
func (t *HDFSCheckTool) Execute(ctx context.Context, input map[string]interface{}) (string, error) {
    // 没有任何实际逻辑
    return `{"status": "healthy", "total_capacity": "100TB", ...}`, nil  // 硬编码 JSON
}
// 其他 20+ 个工具全部是同样的模式
```

**影响**: MCP 工具是这个项目的核心交互层。面试官问"你的 Agent 怎么执行 HDFS 检查的？"——答案是它根本没执行，直接返回假数据。这是致命的。

**问题 2: RAG 三大检索器全部默认 Mock 模式**

```python
# python/src/aiops/rag/dense.py:36-87
# _use_mock 在 milvus_uri 为空时默认 True
# 返回 8 条硬编码的 mock 文档

# python/src/aiops/rag/sparse.py:192-235  
# Mock 搜索直接复用 DenseRetriever 的 MOCK_DOCUMENTS

# python/src/aiops/rag/graph_rag.py:76-153
# 5 条硬编码因果链 + 3 个历史事件 + 静态拓扑图
```

**问题 3: CrossEncoderReranker 完全是假的**

```python
# python/src/aiops/rag/reranker.py:32-54
class CrossEncoderReranker:
    def rerank(self, query: str, documents: List[Document], top_k: int) -> List[Document]:
        # 没有加载任何模型
        # 只是按某种规则排序然后截断
        return sorted(documents, key=lambda d: len(d.content), reverse=True)[:top_k]
```

**影响**: Reranker 是 RAG 管道的质量关键。你声称用了 Cross-Encoder 重排序来提升检索精度，但实际上只是按文档长度排序。

**问题 4: Planning 节点从未调用 RAG**

```python
# python/src/aiops/agent/nodes/planning.py:70-137
def process(self, state: AgentState) -> AgentState:
    # 文档说这个节点会输出 rag_context
    # 但代码中从未调用 HybridRetriever
    # rag_context 字段从未被写入 state
    pass
```

**问题 5: Graph-RAG 未接入 RRF 融合**

```python
# python/src/aiops/rag/retriever.py:84-106
class HybridRetriever:
    def retrieve(self, query, top_k):
        dense_results = self.dense.search(query, top_k)
        sparse_results = self.sparse.search(query, top_k)
        # RRF 融合只用了 Dense + Sparse
        # Graph-RAG 的结果根本没参与融合！
        return self.rrf_fusion(dense_results, sparse_results)
```

**问题 6: Go 端零测试 + 伪 Tracing**

```go
// go/internal/middleware/tracing.go:12-36
// 函数名叫 "tracing middleware"
// 但实际只是 zerolog 打了个日志
// 没有 OTel span、没有 trace ID、没有 context 传播

// Go 目录下 0 个 *_test.go 文件
```

**问题 7: CI/CD 引用不存在的 Dockerfile**

```yaml
# .github/workflows/ci.yml
# 6 阶段 CI/CD pipeline 写得很漂亮
# 但引用的 Dockerfile 不存在于仓库中
```

#### 关键判断

这个项目**概念和架构都很好**，是四个项目中最有"讲故事"潜力的。但当前状态是一个**精心设计的 Demo 骨架**，不是生产级系统。面试时需要诚实地定位为 "PoC / 技术验证项目"，而不是 "生产级项目"。

## 3. 跨项目共性问题

这些是四个项目中反复出现的问题模式，也是面试中最容易被系统性追问的点：

### 3.1 测试荒漠 🏜️

| 项目 | 源文件数 | 测试文件数 | 覆盖率估计 |
|------|---------|-----------|-----------|
| notification-platform | ~280 | ~15（全集成测试） | < 10% |
| distributed_task_platform | 87 | 11 | < 15% |
| ai-dataplatform-ops (Go) | ~30 | 0 | 0% |
| ai-dataplatform-ops (Python) | ~25 | ~8 | ~20%（但多为路由测试） |
| data-platform | ~50 | 0 | 0% |

**面试风险**: "你怎么保证代码质量？" → 答不上来。没有单测 = 没有回归保障 = 不敢重构 = 技术债累积。

### 3.2 错误处理模式混乱

- `_ = err` 静默吞错：4 个项目中出现 20+ 次
- `panic("implement me")`：至少在 2 个项目中出现
- MQ Consumer 启动失败直接 panic（应返回 error）
- 批量操作只保留最后一个 error（前面的丢失）

### 3.3 优雅退出 / 并发安全

- SchedulerV2 的 goroutine 无 WaitGroup 追踪
- Runner goroutine 无追踪，可能泄漏
- 全局 map 无锁保护（data-platform）
- GracefulStop 不等待 goroutine 完成

### 3.4 工程化缺失

| 缺失项 | 涉及项目 |
|--------|---------|
| CI/CD Pipeline | notification-platform, distributed_task_platform, data-platform |
| Dockerfile | notification-platform, distributed_task_platform, data-platform |
| K8s 部署配置 | 全部 |
| 性能压测数据 | 全部 |
| API 文档（Swagger） | 全部 |
| 可观测性（Metrics/Tracing） | 全部（ai-dataplatform-ops 的 tracing 是假的）|

## 4. 市场竞争力分析

### 4.1 你目前的能力画像

```
✅ 已具备（面试可讲）:
   - Go 后端开发（4 年经验）
   - 分布式系统设计（任务调度、分片、DAG）
   - 消息系统（Kafka 消费者、事件驱动）
   - 大数据平台（HDFS/ES/Kafka/Impala）
   - AI Agent 概念（LangGraph/MCP/RAG）

❌ 缺失（面试加分项）:
   - 高并发/高可用的量化证明（QPS 多少？延迟多少？可用性几个 9？）
   - 微服务治理实战（服务发现、熔断、限流、链路追踪）
   - 云原生实践（K8s 部署、Helm Chart、GitOps）
   - 测试体系（单测 + 集成测试 + 压测 = 完整质量保障）
   - 数据库深度（分库分表、读写分离、缓存一致性方案）
```

### 4.2 与市场要求的差距

2026 年后端开发（尤其是 Go 方向）的面试重点：

| 能力维度 | 市场要求 | 你的现状 | 差距 |
|---------|---------|---------|------|
| 系统设计 | 能设计百万级 QPS 系统 | 有架构但无量化数据 | 中 |
| 微服务治理 | 熔断/限流/链路追踪实战 | 无实际实现 | **大** |
| 云原生 | K8s + 容器化 + CI/CD | 几乎为零 | **大** |
| 测试工程 | 70%+ 覆盖率 | < 10% | **大** |
| 数据库 | 分库分表 + 缓存一致性 | 仅基础 CRUD | 中 |
| AI 工程化 | Agent 实际落地 | 全 Mock | 中 |

### 4.3 关键洞察

你的项目覆盖面其实**很广**——任务调度、消息通知、AI Agent、数据平台。但问题是**深度不够**。每个项目都像是写了 70% 就停了：

- 架构设计好了 → 但测试没写
- 调度逻辑实现了 → 但优雅退出有 bug
- AI Agent 框架搭好了 → 但核心能力全是 Mock
- 配置管理做了 → 但密码硬编码在 Git 里

面试官看的不是你"会不会"，而是你"做没做到位"。

## 5. 新项目推荐

### 5.1 推荐方向：高性能 API 网关（Go）

#### 为什么是这个？

| 理由 | 说明 |
|------|------|
| **补齐最大短板** | 微服务治理（限流/熔断/链路追踪）是你四个项目都没覆盖的 |
| **技术深度天花板高** | 涉及高性能网络编程、插件化架构、动态配置热加载 |
| **面试故事性强** | "我自己写了一个 API 网关" ≈ 面试官立刻来兴趣 |
| **与现有项目互补** | 可以作为其他三个项目的统一入口，形成完整的微服务体系 |
| **Go 生态契合** | API 网关是 Go 语言最擅长的领域之一 |

#### 核心功能设计

```
┌─────────────────────────────────────────────────────────┐
```

#### 技术栈建议

| 层面 | 选型 | 理由 |

#### 面试亮点预设

1. **"你的限流怎么做的？"** → 单机令牌桶 + Redis 分布式限流，支持按 API/用户/IP 三个维度，配额可通过 etcd 动态调整
2. **"熔断降级方案？"** → 滑动窗口统计失败率，三态断路器（Closed → Open → Half-Open），支持自定义降级返回
3. **"怎么保证网关本身的高可用？"** → 无状态设计 + K8s 多副本 + 健康检查 + 优雅退出（WaitGroup + context cancel，这次一定写对）
4. **"性能数据？"** → benchmark 证明单机 10w+ QPS，P99 延迟 < 5ms（必须有数据）

### 5.2 备选方向

如果你觉得 API 网关太"基础设施"，以下是其他可以显著提升竞争力的方向：

#### 备选 A: 分布式缓存系统（Mini-Redis）

```
优势: 展示网络编程 + 数据结构 + 并发控制的深度
技术点: RESP 协议解析、内存管理、持久化（AOF/RDB）、集群模式
面试故事: "我用 Go 实现了一个支持集群的 Redis 子集"
```

#### 备选 B: 实时数据管道（Stream Processing）

```
优势: 与大数据背景完美契合，补齐实时计算能力
技术点: Kafka Consumer Group、窗口聚合、Exactly-Once 语义、背压控制
面试故事: "我设计了一个实时指标聚合管道，支持 tumbling/sliding window"
适合度: ★★★★★（与你的大数据背景最匹配）
```

#### 备选 C: 把 ai-dataplatform-ops 做"实"

```
优势: 不需要新开项目，把 Mock 全部替换为真实实现
工作量: 接入真实 Milvus + 真实 LLM API + 真实 MCP 工具
风险: 依赖外部服务，演示和面试时需要环境准备
建议: 至少把 Reranker 和 Dense Retriever 换成真实模型
```

### 5.3 最终建议

**最优策略 = API 网关（新项目） + 修复现有项目（并行推进）**

理由：
1. API 网关覆盖了你最大的能力缺口（微服务治理 + 云原生 + 测试体系）
2. 同时花 1-2 周修复 notification-platform 和 distributed_task_platform 的核心问题
3. data-platform 如果时间不够就从简历上去掉
4. ai-dataplatform-ops 至少把 Reranker 换成真实模型，其他 Mock 可以面试时说"在迭代中"

你的项目组合在**广度上已经很有竞争力**——从底层数据平台到上层 AI Agent，覆盖了后端开发的多个核心领域。但面试官不只看广度，还看深度和完成度。

3. 📊 **用数据说话** —— 每个项目都要有性能数据：QPS、延迟、可用性

做到这三点，你的简历竞争力会有质的飞跃。

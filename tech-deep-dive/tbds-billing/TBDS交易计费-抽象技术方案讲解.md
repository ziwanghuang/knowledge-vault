
# 交易计费系统 —— 抽象技术方案讲解

> 本文档将 TBDS 交易计费系统的业务场景进行抽象简化，剥离具体的 EMR/CVM/CDB 业务细节，以通用的技术语言讲解核心设计方案。适用于面试中用通用技术术语展示分布式系统设计能力。

---

## 一、问题抽象

### 1.1 业务场景抽象

将具体业务抽象为通用的**云资源编排交付系统**：

| 具体业务 | 抽象表述 |
|---------|---------|
| 用户购买 EMR 集群 | 用户下单一个**组合商品**（包含多种异构资源） |
| CVM 云服务器 | 计算资源（Resource Type A） |
| CDB 云数据库 | 存储资源（Resource Type B） |
| CBS 云硬盘 | 附属资源（Resource Type C） |
| EMR 服务费 | 平台服务费（Service Fee） |
| 计费系统回调 | 外部系统异步通知 |
| woodpecker 部署集群 | 下游系统初始化 |

### 1.2 核心技术问题

> **"如何在一次用户操作中，协调多个独立的外部系统完成资源交付，并保证数据一致性？"**

具体拆解为 5 个子问题：

1. **跨系统一致性**：一次操作涉及 3-5 个独立系统（支付、计算资源、存储资源、平台服务），如何保证要么全部成功，要么正确补偿？
2. **异步回调可靠性**：外部系统通过回调通知结果，如何防止重复回调、丢失回调？
3. **部分失败处理**：5 台计算资源只成功了 3 台，如何处理？
4. **长流程状态管理**：从下单到交付可能需要 5-30 分钟，如何管理中间状态？
5. **双模式计费**：预付费（先付款后交付）和后付费（先交付后结算）两种模式如何统一？

## 二、核心技术方案

### 2.1 方案总览：编排式工作流 + 最终一致性

## 三、六大核心技术点详解

### 3.1 编排式工作流引擎

**问题**：一次资源交付涉及 10+ 个步骤，如何可靠地编排执行？

**方案**：基于 BPMN 的异步工作流引擎

```
┌─────────────────────────────────────────────────────┐
│                  工作流引擎架构                        │
│                                                      │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐       │
│  │ Flow     │───>│TaskCenter│───>│TaskHandler│       │
│  │ 定义     │    │ 调度器   │    │ 执行器    │       │
│  └──────────┘    └──────────┘    └──────────┘       │
│                       │                              │
│                       ▼                              │
│              ┌────────────────┐                      │
│              │  三态返回机制   │                      │
│              │                │                      │
│              │  SUCCESS ──────│──> 下一步             │
│              │  IN_PROCESS ──│──> 等待重试            │
│              │  FAIL ────────│──> 补偿/告警           │
│              └────────────────┘                      │
└─────────────────────────────────────────────────────┘
```

**核心设计**：
- **声明式流程定义**：用 BPMN XML 定义步骤顺序和依赖关系
- **三态返回**：每个 TaskHandler 返回 SUCCESS / IN_PROCESS / FAIL
- **轮询调度**：TaskCenter 定时扫描 IN_PROCESS 状态的任务，自动重试
- **超时控制**：每个步骤有独立的超时时间，超时自动标记为 FAIL

**面试话术**：
> "我们设计了一个基于 BPMN 的异步工作流引擎。每个步骤是一个 TaskHandler，返回三种状态：SUCCESS 进入下一步，IN_PROCESS 表示还在处理中（比如等待外部 API 返回），FAIL 进入补偿流程。TaskCenter 作为调度器，定时轮询 IN_PROCESS 的任务，天然支持重试和断点续传。"

### 3.2 分布式锁 + 幂等性设计

**问题**：外部系统可能重复回调（网络超时重试），如何防止重复创建资源？

**方案**：两层防护

```
请求进入
    │
    ▼
┌──────────────────────────┐
│ 第 1 层：Redis 分布式锁   │
│ Key = "lock:{OrderId}"   │
│ TTL = 10s                │
│ 获取失败 → 直接返回       │
└──────────┬───────────────┘
           │ 获取成功
           ▼
┌──────────────────────────┐
│ 第 2 层：FlowId 重入检查  │
│ 查询 DB：该 OrderId       │
│ 是否已有关联的 FlowId？   │
│ 有 → 返回已有 FlowId     │
│ 无 → 创建新工作流         │
└──────────────────────────┘
```

**为什么需要两层？**
- **分布式锁**：防止并发请求（毫秒级重复）
- **FlowId 重入检查**：防止锁释放后的重复请求（秒级/分钟级重复）
- 两者互补，覆盖不同时间窗口的重复场景

**面试话术**：
> "幂等性我们做了两层：第一层是 Redis 分布式锁，以订单号为键，10 秒超时，防止并发重复回调；第二层是数据库级别的 FlowId 重入检查，查询该订单是否已经关联了工作流，如果有就直接返回已有的 FlowId，不重复创建。两层互补，分别覆盖毫秒级和秒级的重复场景。"

### 3.3 资源状态机设计

**问题**：每个资源（计算实例、存储实例）有复杂的生命周期，如何管理？

**方案**：有限状态机（FSM）

```
                    ┌──────────┐
                    │  INIT(0) │
                    └────┬─────┘
                         │ 订单创建
                         ▼
                  ┌──────────────┐
                  │ POLL_WAIT(1) │
                  └──────┬───────┘
                         │ 资源就绪
            ┌────────────┼────────────┐
            │            │            │
            ▼            ▼            ▼
     ┌────────────┐ ┌─────────┐ ┌──────────┐
     │RENEW_WAIT  │ │ READY(2)│ │POST_PUSH │
     │续费等待     │ │ 就绪可用 │ │用量上报   │
     └────────────┘ └────┬────┘ └──────────┘
                         │
              ┌──────────┼──────────┐
              │          │          │
              ▼          ▼          ▼
       ┌───────────┐ ┌────────┐ ┌──────────┐
       │UPDATE_WAIT│ │ISOLATED│ │DESTROY   │
       │变配等待    │ │已隔离   │ │WAIT      │
       └───────────┘ └───┬────┘ └────┬─────┘
                         │           │
                         ▼           ▼
                    ┌─────────┐ ┌──────────┐
                    │ 续费找回 │ │DESTROYED │
                    │→ READY  │ │已销毁     │
                    └─────────┘ └──────────┘
```

**核心设计**：
- **26 种状态**覆盖完整生命周期（创建/就绪/续费/变配/隔离/销毁）
- **核心路径只有 7 种**：INIT → POLL_WAIT → READY → ISOLATED → DESTROYED
- **中间态**（如 RENEW_WAIT、UPDATE_WAIT）标记当前正在执行的操作类型
- **大小订单关联**：大订单代表整个组合商品，小订单代表每个子资源

**面试话术**：
> "资源状态机有 26 种状态，但核心路径只有 7 种。中间态用于标记当前正在执行的操作类型，比如 RENEW_WAIT 表示正在续费中。通过大小订单的关联关系，一个大订单可以管理多个异构资源的独立生命周期。"

### 3.4 策略模式处理双模式计费

**问题**：预付费和后付费两种模式，流程差异大，如何避免代码重复？

**方案**：策略模式 + 模板方法

```
┌─────────────────────────────────────────┐
│          BillingStrategy 接口            │
│  ┌─────────────────────────────────┐    │
│  │ + CreateResource()              │    │
│  │ + RenewResource()               │    │
│  │ + IsolateResource()             │    │
│  │ + DestroyResource()             │    │
│  │ + GetResourceStatus()           │    │
│  └─────────────────────────────────┘    │
│           ▲              ▲               │
│           │              │               │
│  ┌────────┴───┐  ┌──────┴────────┐     │
│  │ PrepayMode │  │ PostpayMode   │     │
│  │ 预付费策略  │  │ 后付费策略    │     │
│  │            │  │               │     │
│  │ 支付回调   │  │ 直接创建      │     │
│  │ 触发交付   │  │ 定时上报用量  │     │
│  │ 到期隔离   │  │ 欠费隔离      │     │
│  └────────────┘  └───────────────┘     │
└─────────────────────────────────────────┘
```

**核心差异**：

| 维度 | 预付费策略 | 后付费策略 |
|------|-----------|-----------|
| 触发方式 | 外部系统支付回调 | 用户直接请求 |
| 资源创建 | 外部系统下单生产 | 直接调用资源 API |
| 计费方式 | 一次性支付 | 定时上报用量，按时结算 |
| 欠费处理 | 到期隔离 | 欠费隔离 → 冲正恢复 / 超时销毁 |

**面试话术**：
> "预付费和后付费通过策略模式统一。两种模式共享同一套工作流引擎和状态机，差异点通过策略接口隔离：预付费由支付回调触发，后付费由用户直接请求触发；预付费一次性支付，后付费定时上报用量。这样核心的资源交付、状态管理、补偿逻辑只需要写一份。"

### 3.5 分级补偿策略

**问题**：10 步工作流中，任何一步都可能失败，如何处理？

**方案**：按失败阶段分级补偿

```
┌─────────────────────────────────────────────────────┐
│              分级补偿策略                              │
│                                                      │
│  ┌─────────────────────────────────────────────┐    │
│  │ Level 1：自动重试（可恢复错误）               │    │
│  │ ─────────────────────────────────────────── │    │
│  │ 网络超时、API 限流、临时不可用                 │    │
│  │ → TaskHandler 返回 IN_PROCESS                │    │
│  │ → TaskCenter 自动重试（指数退避）             │    │
│  │ 适用步骤：所有步骤                            │    │
│  └─────────────────────────────────────────────┘    │
│                                                      │
│  ┌─────────────────────────────────────────────┐    │
│  │ Level 2：部分退款（部分成功）                  │    │
│  │ ─────────────────────────────────────────── │    │
│  │ 5 台资源只成功了 3 台                         │    │
│  │ → 自动触发部分退款流程                        │    │
│  │ → 退还失败资源的费用                          │    │
│  │ → 已成功的资源继续交付                        │    │
│  │ 适用步骤：资源申请阶段                        │    │
│  └─────────────────────────────────────────────┘    │
│                                                      │
│  ┌─────────────────────────────────────────────┐    │
│  │ Level 3：全额退款（全部失败）                  │    │
│  │ ─────────────────────────────────────────── │    │
│  │ 资源申请全部失败                              │    │
│  │ → 触发全额退款流程                            │    │
│  │ → 释放所有已申请的资源                        │    │
│  │ → 订单状态标记为失败                          │    │
│  │ 适用步骤：资源申请阶段                        │    │
│  └─────────────────────────────────────────────┘    │
│                                                      │
│  ┌─────────────────────────────────────────────┐    │
│  │ Level 4：自动销毁 + 退款（初始化失败）         │    │
│  │ ─────────────────────────────────────────── │    │
│  │ 资源已申请成功，但下游系统初始化失败            │    │
│  │ → 根据策略决定：自动销毁 or 挂起等待           │    │
│  │ → 销毁所有已创建的资源                        │    │
│  │ → 触发退款流程                               │    │
│  │ 适用步骤：下游系统初始化阶段                   │    │
│  └─────────────────────────────────────────────┘    │
│                                                      │
│  ┌─────────────────────────────────────────────┐    │
│  │ Level 5：挂起 + 告警 + 人工介入               │    │
│  │ ─────────────────────────────────────────── │    │
│  │ 不可恢复的异常（数据不一致、未知错误）          │    │
│  │ → 流程挂起，不再自动重试                      │    │
│  │ → 发送告警通知运维人员                        │    │
│  │ → 人工排查后手动恢复或取消                    │    │
│  │ 适用步骤：所有步骤的兜底策略                   │    │
│  └─────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────┘
```

**核心原则**：
- **越早失败，处理越简单**（自动重试即可）
- **越晚失败，处理越复杂**（需要退款、销毁等补偿操作）
- **涉及金额的操作，宁可挂起也不自动补偿**（人工确认更安全）

**面试话术**：
> "失败处理我们做了分级补偿：第一级是自动重试，网络抖动等可恢复错误直接重试；第二级是部分退款，比如 5 台机器只成功了 3 台，自动退还失败部分的费用；第三级是全额退款，资源申请全部失败时触发；第四级是自动销毁加退款，资源已创建但初始化失败时销毁所有资源；第五级是挂起告警，不可恢复的异常交给人工处理。核心原则是：越早失败处理越简单，涉及金额的操作宁可挂起也不自动补偿。"

### 3.6 定时对账机制

**问题**：如何发现系统中的数据不一致？

**方案**：双向定时对账

```
┌──────────────────────────────────────────┐
│            定时对账机制                     │
│                                           │
│  ┌─────────────────────────────────┐     │
│  │ 内部对账（我方主动）              │     │
│  │ ─────────────────────────────── │     │
│  │ 定时任务：每天凌晨执行            │     │
│  │ 逻辑：                           │     │
│  │   1. 查询所有活跃订单             │     │
│  │   2. 对每个订单调用实时询价接口    │     │
│  │   3. 对比订单实际价格 vs 询价结果  │     │
│  │   4. 不一致 → 告警 + 记录日志     │     │
│  └─────────────────────────────────┘     │
│                                           │
│  ┌─────────────────────────────────┐     │
│  │ 外部对账（对方巡检）              │     │
│  │ ─────────────────────────────── │     │
│  │ 计费系统每天巡检所有资源状态       │     │
│  │ 逻辑：                           │     │
│  │   1. 查询所有已发货资源           │     │
│  │   2. 调用我方接口确认资源存活     │     │
│  │   3. 发现"已发货但资源不存在"    │     │
│  │   4. 触发异常处理流程             │     │
│  └─────────────────────────────────┘     │
│                                           │
│  双向对账 = 我方主动 + 对方巡检            │
│  确保任何一方的数据异常都能被发现           │
└──────────────────────────────────────────┘
```

**面试话术**：
> "对账我们做了双向机制：内部对账是我们每天定时拿订单价格和实时询价结果对比，发现不一致就告警；外部对账是计费系统每天巡检所有已发货资源的存活状态。双向对账确保任何一方的数据异常都能被发现。"

## 四、为什么不用 Saga / TCC？

这是面试中最有区分度的问题。

### 4.1 评估过程

| 方案 | 评估结论 | 原因 |
|------|---------|------|
| **TCC** | ❌ 不适用 | 外部资源 API 不支持 Try/Confirm/Cancel 三阶段语义 |
| **标准 Saga** | ❌ 不适用 | 补偿操作本身也可能失败（"销毁资源"也可能超时） |
| **事务消息** | ❌ 不适用 | 不是简单的消息通知场景，是复杂的多步编排 |
| **编排式工作流** | ✅ 采用 | 天然支持异步、重试、部分失败，且不依赖外部框架 |

### 4.2 三个核心原因

**原因 1：外部 API 不支持补偿语义**
> "购买资源"的补偿是"销毁资源"，但销毁本身也可能失败。标准 Saga 要求补偿操作是可靠的，但在云 API 场景下无法保证。

**原因 2：业务天然是异步的**
> 用户创建一个资源组合本来就要等几分钟到几十分钟，不需要同步强一致性。异步工作流 + 轮询重试天然契合这个场景。

**原因 3：涉及金额的操作需要人工确认**
> 自动补偿可能导致"多退少补"的资金风险。出现异常时挂起流程让人工确认，比自动补偿更安全。

### 4.3 面试话术

> "我们评估过 Saga 和 TCC，但最终没有采用。三个原因：
> 1. 外部云 API 不支持标准的补偿语义——'购买资源'的补偿是'销毁资源'，但销毁本身也可能失败，不满足 Saga 对补偿操作可靠性的要求；
> 2. 业务天然是异步的——用户创建资源组合本来就要等几分钟，不需要同步强一致性，异步工作流 + 轮询重试天然契合；
> 3. 涉及金额的操作——出现异常时挂起流程让人工确认，比自动补偿更安全。
>
> 所以我们设计了一套编排式工作流方案，本质上是 Saga 编排模式的一种工程实现，但更务实——用三态返回 + 轮询调度替代了标准 Saga 的补偿链，用分级补偿替代了统一的回滚策略。"

## 五、面试完整讲述模板

### 5.1 一句话概述（10 秒）

> "我负责设计了一套云资源编排交付系统，支持预付费和后付费两种模式，通过异步工作流 + 状态机 + 分级补偿保证跨系统的最终一致性。"

### 5.2 展开讲解（2 分钟）

> "核心场景是：用户下单一个组合商品，系统需要协调多个独立的外部系统完成资源交付。
>
> 我设计了四层防线保证一致性：
> 1. **本地事务**保证单库原子性
> 2. **异步状态机**驱动流程推进——TaskHandler 返回三态（成功/处理中/失败），调度器轮询驱动，天然支持重试
> 3. **分布式锁 + 幂等**防止重复操作——Redis 锁防并发，FlowId 重入检查防重复
> 4. **定时对账**兜底——双向对账发现数据不一致
>
> 失败处理采用分级补偿：可恢复错误自动重试，部分失败自动部分退款，不可恢复错误挂起告警人工介入。"

### 5.3 技术亮点清单

| 技术点 | 一句话描述 | 面试价值 |
|--------|-----------|---------|
| 编排式工作流 | 基于 BPMN 的异步状态机，三态返回 + 轮询调度 | ⭐⭐⭐⭐⭐ 核心亮点 |
| 分布式锁 + 幂等 | Redis 锁 + FlowId 重入检查，双层防护 | ⭐⭐⭐⭐ 高频考点 |
| 分级补偿策略 | 5 级补偿，越早失败处理越简单 | ⭐⭐⭐⭐ 体现工程判断力 |
| 策略模式 | 预付费/后付费通过策略接口隔离 | ⭐⭐⭐ 设计模式应用 |
| 资源状态机 | 26 种状态，核心路径 7 种 | ⭐⭐⭐ 复杂状态管理 |
| 定时对账 | 双向对账，价格校验 + 状态巡检 | ⭐⭐⭐ 数据一致性兜底 |
| 不用 Saga 的决策 | 三个原因：API 不支持补偿、天然异步、金额安全 | ⭐⭐⭐⭐⭐ 最大区分度 |

## 六、与业界方案的对应关系

| 业界方案 | 我们的实现 | 对应关系 |
|---------|-----------|---------|
| Saga 编排式 | 异步工作流 + 三态返回 | 本质相同，但不依赖标准 Saga 框架 |
| 补偿事务 | 分级补偿策略 | 更细粒度，按失败阶段分级处理 |
| 幂等性设计 | 分布式锁 + FlowId 重入 | 双层防护，覆盖不同时间窗口 |
| 最终一致性 | 轮询重试 + 定时对账 | 异步驱动 + 兜底对账 |
| 状态机模式 | 资源状态机（26 种状态） | 标准 FSM 实现 |
| 策略模式 | 预付费/后付费策略 | 标准设计模式应用 |

> **核心定位**：我们的方案是 Saga 编排模式的一种**工程化实现**，针对"外部 API 不支持补偿语义"的现实约束做了务实的适配。

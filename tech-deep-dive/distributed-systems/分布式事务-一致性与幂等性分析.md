# 分布式事务、一致性与幂等性分析报告

> 目标：梳理 `data-platform` 和 `distributed_task_platform` 中所有需要分布式事务、一致性保障和幂等性的场景，给出改进方案和面试话术。

---

## 一、distributed_task_platform

### 1. 终态处理四步非原子操作 🔴 高危

**文件**: `internal/service/task/execution_service.go:305-322`

**现状**:
```go
// 4 个步骤，顺序执行，无事务包裹
s.executionRepo.UpdateState(ctx, execution)   // 1. 更新执行记录状态
s.taskAcquirer.Release(ctx, task)             // 2. 释放任务（CAS）
s.taskRepo.UpdateNextTime(ctx, task)          // 3. 更新下次调度时间
s.producer.Send(ctx, completeEvent)           // 4. 发送 MQ 完成事件
```

**风险**: 步骤 1 成功但步骤 2 失败 → 任务永远被锁定，不会再被调度。步骤 1-3 成功但步骤 4 失败 → 下游（如 DAG 编排）永远收不到完成通知。

**改进方案 — Transactional Outbox**:
```go
func (s *ExecutionService) handleTerminalStatus(ctx context.Context, ...) error {
    return s.db.Transaction(func(tx *gorm.DB) error {
        // 1-3 在同一个事务中
        if err := tx.Model(&execution).Updates(...); err != nil {
            return err
        }
        if err := tx.Model(&task).Updates(map[string]interface{}{
            "status": "IDLE", "version": gorm.Expr("version + 1"),
        }).Where("version = ?", task.Version).Error; err != nil {
            return err // CAS 失败回滚
        }
        if err := tx.Model(&task).Update("next_time", nextTime).Error; err != nil {
            return err
        }
        // 4. 写入 outbox 表（同事务）
        return tx.Create(&OutboxMessage{
            Topic:   "task.complete",
            Payload: marshal(completeEvent),
            Status:  "PENDING",
        }).Error
    })
    // 异步 goroutine 轮询 outbox 表发送 MQ
}
```

**面试话术**: "任务终态处理涉及状态更新、锁释放、调度时间更新和 MQ 通知四个步骤。我用 **Transactional Outbox 模式** 将前三步和 MQ 消息写入放在同一个数据库事务中，由后台 relay 协程异步投递消息，保证了 **本地事务 + 最终一致性**。Outbox 表带 `created_at` 索引和 TTL 清理，relay 使用 `SELECT ... FOR UPDATE SKIP LOCKED` 避免多实例争抢。"

### 2. 分片任务创建非原子 🔴 高危

**文件**: `internal/service/task/execution_service.go:148-156`

**现状**:
```go
parent, err := s.executionRepo.CreateShardingParent(ctx, ...)  // 创建父记录
if err != nil { return err }
err = s.executionRepo.CreateShardingChildren(ctx, children)     // 创建子记录（批量）
// 父成功子失败 → 孤儿父记录，ShardingCompensator 兜底但浪费一个调度周期
```

**改进方案 — 本地事务包裹**:
```go
err := s.db.Transaction(func(tx *gorm.DB) error {
    parent := &TaskExecution{...ShardingParentID: 0...}
    if err := tx.Create(parent).Error; err != nil {
        return err
    }
    for i := range children {
        children[i].ShardingParentID = parent.ID
    }
    return tx.CreateInBatches(children, 100).Error
})
```

**面试话术**: "分片任务的父子记录创建必须原子化。我把它们放进同一个 MySQL 事务，避免了孤儿记录问题。之前虽然有 ShardingCompensator 做补偿，但那是 **被动兜底**，改成事务后是 **主动预防**，补偿器变成了真正的 last resort。"

### 3. 抢占 + 执行记录创建的中间态 🟡 中危

**文件**: `internal/service/runner/normal_task_runner.go:103-139`

**现状**:
```go
task, err := s.acquirer.Acquire(ctx, ...)       // CAS 抢占任务
execution, err := s.executionRepo.Create(ctx, ...)  // 创建执行记录
go func() {
    result := s.invoker.Run(ctx, ...)            // 异步执行
    if err != nil {
        // 仅日志，不更新执行记录状态！
        logger.Error("invoke failed", ...)
    }
}()
```

**风险**:
1. 抢占成功但创建执行记录失败 → `releaseTask` 可能也失败 → 任务永久卡死
2. `invoker.Run` 失败时执行记录停留在 PREPARE 状态，但 `InterruptCompensator` 只扫描 RUNNING 状态 → PREPARE 记录永远不会被清理

**改进方案**:
```go
// 1. Acquire + Create 放进事务（需要 repository 支持 tx 参数）
err := s.db.Transaction(func(tx *gorm.DB) error {
    if err := acquireWithTx(tx, task); err != nil {
        return err
    }
    return createExecutionWithTx(tx, execution)
})

// 2. invoker.Run 失败时必须更新执行记录
go func() {
    result := s.invoker.Run(ctx, ...)
    if err != nil {
        s.executionRepo.UpdateState(ctx, execution.ID, FAILED, err.Error())
        s.acquirer.Release(ctx, task) // 释放任务
    }
}()

// 3. InterruptCompensator 同时扫描 PREPARE 和 RUNNING
```

**面试话术**: "任务抢占和执行记录创建之间存在中间态风险。我做了三件事：(1) 用事务保证原子性；(2) 异步执行失败时主动更新状态并释放任务，避免 '哑弹' 记录；(3) 补偿器扩展到覆盖 PREPARE 状态。这是 **防御性编程 + 补偿机制** 的组合。"

### 4. MQ 消费无幂等保障 🔴 高危

**文件**: `internal/event/reportevt/report_event_consumer.go:56-87`，`batch_report_event_consumer.go:47-83`

**现状**:
```go
func (c *Consumer) consumeExecutionReportEvent(msg *sarama.ConsumerMessage) error {
    report := unmarshal(msg.Value)
    return c.executionService.HandleReports(ctx, report)  // 直接处理，无去重
}
```

**风险**: Kafka rebalance 或 consumer 重启导致消息重投 → 同一条 report 被处理两次。终态状态因为有 `isTerminalStatus` 检查有部分保护，但非终态（如 RUNNING 进度更新）会被重复处理。

**改进方案 — 幂等消费**:
```go
// 方案 A：业务层幂等（推荐）
func (s *ExecutionService) HandleReports(ctx context.Context, reports []*Report) error {
    for _, r := range reports {
        // 使用 (execution_id, sequence_no) 做幂等键
        affected := tx.Model(&TaskExecution{}).
            Where("id = ? AND last_sequence < ?", r.ExecutionID, r.SequenceNo).
            Updates(map[string]interface{}{
                "progress": r.Progress,
                "last_sequence": r.SequenceNo,
            }).RowsAffected
        if affected == 0 {
            continue // 已处理过，跳过
        }
    }
}

// 方案 B：去重表
type ConsumedMessage struct {
    MessageKey string `gorm:"primaryKey"` // topic:partition:offset
    ConsumedAt time.Time
}
```

**面试话术**: "MQ 消费我用了两层幂等保障：(1) 业务层，给执行记录加 `last_sequence` 字段，用单调递增的序号做 **条件更新**（UPDATE WHERE sequence < new_sequence），天然过滤重复和乱序消息；(2) 基础设施层，用消费位点（topic:partition:offset）做去重表兜底。Kafka 的 at-least-once 语义下，**消费端幂等是必须的**。"

### 1. Job 创建缺幂等键 🟡 中危

**文件**: `tbds-control/internal/server/api/job_handler.go:64, 78-118`

**现状**:
```go
processID := fmt.Sprintf("%d", time.Now().UnixMilli())  // 毫秒时间戳做 ID，并发冲突

err := db.DB.Transaction(func(tx *gorm.DB) error {  // ✅ 事务做得好
    tx.Create(job)
    for _, stage := range stages {
        tx.Create(stage)
    }
    return nil
})
// 但无幂等键：前端重试 → 重复创建 Job
```

**改进方案**:
```go
// 1. processID 用 UUID 或 snowflake
processID := uuid.New().String()

// 2. 加幂等键
type Job struct {
    // ...
    IdempotencyKey string `gorm:"uniqueIndex;size:64"` // 客户端生成
}

// 3. 创建时带 ON CONFLICT 检查
func CreateJob(tx *gorm.DB, job *Job) error {
    result := tx.Clauses(clause.OnConflict{
        Columns:   []clause.Column{{Name: "idempotency_key"}},
        DoNothing: true,
    }).Create(job)
    if result.RowsAffected == 0 {
        // 返回已存在的 Job
        return tx.Where("idempotency_key = ?", job.IdempotencyKey).First(job).Error
    }
    return result.Error
}
```

**Schema 变更**:
```sql
ALTER TABLE job ADD COLUMN idempotency_key VARCHAR(64) NOT NULL DEFAULT '';
CREATE UNIQUE INDEX idx_job_idempotency ON job(idempotency_key);
```

**面试话术**: "Job 创建接口我做了幂等改造：客户端在请求头带 `X-Idempotency-Key`（前端用 UUID），服务端对该字段建唯一索引，INSERT 时用 `ON CONFLICT DO NOTHING`。重复请求直接返回已创建的 Job，**零副作用**。这个模式参考了 Stripe API 的幂等设计。"

### 2. JobWorker 内存缓存与 DB 标记非原子 🔴 高危

**文件**: `tbds-control/internal/server/dispatcher/job_worker.go:50-71`

**现状**:
```go
func (w *JobWorker) syncJobs() {
    jobs := queryUnsyncedJobs()
    for _, job := range jobs {
        w.memStore.SaveJob(job)                          // 写内存
        db.DB.Model(job).Update("synced", models.JobSynced)  // 标记已同步
        // 崩溃发生在这两步之间 → Job 标记为 synced 但内存中没有 → Job 永久丢失
    }
}
```

**改进方案**:
```go
func (w *JobWorker) syncJobs() {
    jobs := queryUnsyncedJobs()
    for _, job := range jobs {
        // 1. 先写内存
        w.memStore.SaveJob(job)
        // 2. 再标记 DB（用 CAS 防并发）
        result := db.DB.Model(job).
            Where("synced = ?", models.JobUnsynced).  // CAS 条件
            Update("synced", models.JobSynced)
        if result.RowsAffected == 0 {
            // 其他实例已处理，从内存移除
            w.memStore.RemoveJob(job.ID)
        }
    }
    // 关键：即使崩溃，MemStoreRefresher 会重新从 DB 加载
    // 但需要确保 synced 标记在崩溃后能被重置
}
```

更彻底的方案：**去掉 synced 标记，改用 MemStoreRefresher 定期全量同步**。synced 字段本身就是分布式状态的一个陷阱。

**面试话术**: "这是一个经典的 **双写一致性** 问题——内存和数据库之间没有事务保障。我的改法是：采用 **DB 为 source of truth** 的策略，去掉 synced 标记，由 Refresher 定期从 DB 拉取状态。内存缓存只做加速，不做持久化保证。这遵循了 '**状态只存一处**' 的设计原则。"

### 3. StageWorker Task 生成无事务 🔴 高危

**文件**: `tbds-control/internal/server/dispatcher/stage_worker.go:54-89`

**现状**:
```go

        task := &models.Task{
            TaskId: fmt.Sprintf("%d", time.Now().UnixMilli()),  // ❌ 循环内毫秒 ID，必然冲突！
            // ...
```

**改进方案**:
```go
func (w *StageWorker) processStage(stage *models.Stage) {
    err := db.DB.Transaction(func(tx *gorm.DB) error {
        tasks := make([]*models.Task, 0, len(stage.Tasks))
        for i, taskDef := range stage.Tasks {
            task := &models.Task{
                TaskId: fmt.Sprintf("%s-%d", stage.StageId, i), // 确定性 ID
                // ...
            }
            tasks = append(tasks, task)
        }
        // 批量创建，全成功或全回滚
        if err := tx.CreateInBatches(tasks, 100).Error; err != nil {
            return err
        }
        // Stage 状态更新也在同一事务中
        return tx.Model(stage).Update("state", models.StageRunning).Error
    })
    if err == nil {
        for _, task := range tasks {
            w.memStore.EnqueueTask(task)
        }
    }
}
```

**面试话术**: "Task 生成有两个问题：一是用毫秒时间戳做 ID 在循环内必然冲突，二是逐条 INSERT 无事务保护。我改为 **确定性 ID**（StageId + 序号）+ **批量事务创建**，保证了原子性。同时 ID 的确定性天然带来了幂等——重试创建同一批 Task 不会产生重复记录。"

### 4. TaskWorker Action 创建 + 状态更新非原子 🔴 高危

**文件**: `tbds-control/internal/server/dispatcher/task_worker.go:91-102`

**现状**:
```go
db.DB.CreateInBatches(actions, 100)                                  // 创建 Actions
db.DB.Model(task).Updates(map[string]interface{}{"state": "Running"}) // 更新 Task 状态
// 两步非原子：Actions 创建成功，状态更新失败
// → Task 留在 Init 状态 → MemStoreRefresher 重新入队 → 再次创建 Actions（重复！）
```

**改进方案**:
```go
err := db.DB.Transaction(func(tx *gorm.DB) error {
    if err := tx.CreateInBatches(actions, 100).Error; err != nil {
        return err
    }
    // 用 CAS 防止并发重复处理
    result := tx.Model(task).
        Where("state = ?", models.TaskInit).  // 只有 Init 状态才更新
        Updates(map[string]interface{}{"state": models.TaskRunning})
    if result.RowsAffected == 0 {
        return fmt.Errorf("task already processed")  // 触发回滚
    }
    return nil
})
```

**面试话术**: "Action 创建和 Task 状态变更必须原子化。我用事务包裹并加了 **CAS 条件**（WHERE state = 'Init'），这样即使 Refresher 重新入队了同一个 Task，CAS 条件不满足会回滚，避免重复创建 Action。这是 **乐观锁 + 事务** 的经典组合。"

### 5. completeStage 级联更新非原子 🔴 高危

**文件**: `tbds-control/internal/server/dispatcher/task_center_worker.go:169-196`

**现状**:
```go
func (w *TaskCenterWorker) completeStage(stage *models.Stage) {
    db.DB.Model(stage).Update("state", models.StageSuccess)     // 标记当前 Stage 完成
    // ---- 如果这里崩溃，下一个 Stage 永远不会被激活 ----
    nextStage := getNextStage(stage.JobId, stage.Order + 1)
    db.DB.Model(nextStage).Update("state", models.StageInit)    // 激活下一个 Stage
    w.memStore.EnqueueStage(nextStage)
}
```

**改进方案**:
```go
func (w *TaskCenterWorker) completeStage(stage *models.Stage) error {
    return db.DB.Transaction(func(tx *gorm.DB) error {
        // 1. CAS 更新当前 Stage
        result := tx.Model(stage).
            Where("state = ?", models.StageRunning).
            Update("state", models.StageSuccess)
        if result.RowsAffected == 0 {
            return nil // 已处理，幂等返回
        }
        // 2. 激活下一个 Stage（同事务）
        nextStage := &models.Stage{}
        if err := tx.Where("job_id = ? AND `order` = ?", 
            stage.JobId, stage.Order+1).First(nextStage).Error; err != nil {
            if errors.Is(err, gorm.ErrRecordNotFound) {
                // 最后一个 Stage，标记 Job 完成
                return tx.Model(&models.Job{}).Where("id = ?", stage.JobId).
                    Update("state", models.JobSuccess).Error
            }
            return err
        }
        return tx.Model(nextStage).Update("state", models.StageInit).Error
    })
}
```

**面试话术**: "Stage 级联推进是整个调度引擎最关键的状态转换。我用 **单数据库事务** 保证当前 Stage 完成和下一个 Stage 激活的原子性，加了 CAS 条件实现幂等。如果是最后一个 Stage，同事务内标记 Job 完成。这消除了 '**Stage 间断裂**' 的风险——之前崩溃在两步之间会导致 Job 永久挂起。"

## 三、通用改进：Transactional Outbox 方案

两个项目中都有 "本地操作成功 + MQ 投递" 的需求，统一用 Outbox 模式解决。

### Outbox 表设计
```sql
CREATE TABLE outbox_messages (
    id          BIGINT AUTO_INCREMENT PRIMARY KEY,
    topic       VARCHAR(128) NOT NULL,
    message_key VARCHAR(128) DEFAULT NULL,
    payload     TEXT NOT NULL,
    status      ENUM('PENDING', 'SENT', 'FAILED') DEFAULT 'PENDING',
    retry_count INT DEFAULT 0,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX idx_status_created (status, created_at)
) ENGINE=InnoDB;
```

### Relay 组件
```go
func (r *OutboxRelay) Run(ctx context.Context) {
    ticker := time.NewTicker(500 * time.Millisecond)
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            r.processMessages(ctx)
        }
    }
}

func (r *OutboxRelay) processMessages(ctx context.Context) {
    var messages []OutboxMessage
    // SELECT ... FOR UPDATE SKIP LOCKED，多实例安全
    r.db.Raw(`SELECT * FROM outbox_messages 
              WHERE status = 'PENDING' AND retry_count < 5
              ORDER BY created_at LIMIT 50 FOR UPDATE SKIP LOCKED`).
        Scan(&messages)
    
    for _, msg := range messages {
        err := r.producer.Send(ctx, msg.Topic, msg.MessageKey, msg.Payload)
        if err != nil {
            r.db.Model(&msg).Updates(map[string]interface{}{
                "retry_count": gorm.Expr("retry_count + 1"),
                "status":      "FAILED",
            })
        } else {
            r.db.Delete(&msg) // 或标记为 SENT
        }
    }
}
```

**面试话术**: "两个项目都有 DB + MQ 双写的场景。我抽象了一个 **通用 Outbox 组件**：业务方只需在事务中调用 `outbox.Enqueue(tx, topic, payload)`，Relay 协程用 `FOR UPDATE SKIP LOCKED` 竞争消费，支持多实例部署。消息投递做到了 **at-least-once**，消费端配合幂等保证 **exactly-once 语义**。"

## 五、面试整体叙事框架

> **问题引入**: "这两个项目在初版开发时更关注功能完整性，但在压测和线上运行中，我发现了一系列一致性问题……"
>
> **分析方法**: "我从三个维度系统性地审查了代码：(1) 多步操作的原子性——是否需要事务？(2) 失败后的幂等性——重试安全吗？(3) 跨组件的最终一致性——DB 和 MQ 之间靠什么保证？"
>
> **解决策略**: "我没有引入分布式事务中间件（如 Seata），因为这两个系统的跨服务事务场景不多，核心问题都在 **单服务内的多步数据库操作**。我的策略是：**本地事务兜原子性、CAS 兜并发安全、Outbox 兜跨组件一致性、幂等键兜重试安全**。"
>
> **技术选型理由**: "Transactional Outbox 而不是 CDC（如 Debezium），因为项目规模不需要额外组件；CAS（乐观锁）而不是 SELECT FOR UPDATE（悲观锁），因为冲突概率低、吞吐要求高；确定性 ID 而不是 UUID，因为可读性和幂等性更好。"

# 三项目跨项目实现顺序规划

> 基于 2026 年 AI 市场热度和面试 ROI，规划三项目之间的实现优先级
>
> 创建时间：2026-04-15

---

## 一、核心思路

**面试考的是"你能讲多深"，不是"你写了多少行代码"。**

- DTP 和 AIOps 已 100% 完成，重复造轮子（如在 DP 再写 DAG/WAL）ROI 极低
- DP 只需补到"端到端可用 + 2-3 个高价值模块"就够
- AI 工程化是 2026 年最大差异化武器，AIOps 打磨优先于 DP 补全

---

## 二、实现顺序

### 第一优先级：AIOps 打磨（D1-2）

> 代码已 100%，不写新代码，确保能跑能演示

| 事项 | 说明 |
|------|------|
| 本地环境验证 | docker-compose up 一键启动，确认所有服务正常 |
| 端到端流程走通 | 创建对话 → Agent 路由 → 工具调用 → RAG 检索 → 返回结果 |
| 准备 Demo 场景 | 2-3 个典型场景（集群扩容排查、日志异常分析、配置变更审查） |
| 面试话术对齐 | 确保每个模块的设计决策都能讲清楚 why |

**为什么排第一：**

1. **市场热度**：2026 年 AI Agent/LLM 应用是面试最热话题，面试官兴趣远高于传统调度引擎
2. **稀缺性最强**：Go 后端出身能做 LangGraph + MCP + RAG 完整系统的人极少
3. **投入最小**：0 天新代码，1-2 天打磨即可达到演示级别
4. **收益最大**：AI 创业公司（MiniMax/智谱等）和大厂 AI 平台方向对此最感兴趣

## 五、决策依据

### 为什么 AIOps 打磨排在 DP 补全之前？

| 维度 | AIOps 先 | DP 先 |
|------|---------|-------|
| 市场需求匹配度 | ⭐⭐⭐⭐⭐（AI 方向持续升温） | ⭐⭐⭐（大数据平台需求平稳） |
| 投入产出比 | 1-2 天 → 完整可演示 | 5 天 → 才跑通端到端 |
| 差异化程度 | Go 后端 + AI 工程化 = 极稀缺 | 大数据管控 = 有竞争但不稀缺 |
| 面试官兴趣 | 高（新鲜、想聊） | 中（传统、能聊但不兴奋） |
| 对 Offer 的影响 | AI 创业公司 85-90% | 基础设施团队 70-80% |

**结论**：AIOps 是你的"王牌"，必须先确保它能完美展示。DP 是"底牌"，补到够用即可。

### 为什么砍掉 DP 的 DAG/WAL/性能优化？

**避免三个项目之间的能力重复证明**：

| 能力 | 已被哪个项目证明 | DP 不再需要 |
|------|----------------|------------|
| DAG 调度 | DTP（ANTLR4 DSL + Kahn 拓扑） | DP Stage DAG |
| 崩溃恢复 | DTP（四重补偿器） | DP Agent WAL |
| 可观测性 | AIOps（OTel + LangFuse） | DP Prometheus |
| 事件驱动 | DTP（Kafka） | DP Kafka 改造 |

**一句话：三个项目是组合拳，不是三个独立项目各自补全。**

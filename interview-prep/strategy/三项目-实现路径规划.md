## 三、实现路径规划

### 核心原则

1. **先跑通端到端，再加深度** — DP 先完成 Step 4-6 让系统完整可用
2. **按面试价值排序** — 高面试价值的模块优先（Saga > DAG > WAL > 幂等）
3. **每天可验证** — 每个实现阶段都能 `go run` 看到效果
4. **团队建设穿插** — 在 DP 开发间隙完成小 demo，避免疲劳

---

### Phase 1：DP 端到端跑通（第 1-5 天）

**目标**：从"只能创建 Job"到"创建 Job → Agent 执行命令 → 完成"的完整链路

| 天数 | 任务 | 产出 | 验证方式 |
|:----:|------|------|---------|
| **D1** | Step 4: Redis Action 下发 | `internal/server/action/redis_action_loader.go` | Redis CLI 能看到 Action SortedSet |
| **D2** | Step 5: gRPC 服务 + Proto 定义 | `internal/server/grpc/` + `.proto` 文件 | Agent 能 gRPC 连接到 Server |
| **D3** | Step 6: Agent CmdModule + WorkPool | `internal/agent/cmd_module.go` + `work_pool.go` | Agent 拉取到 Action 并执行 Shell |
| **D4** | Step 6: Agent 结果上报 + Server 处理 | 上报链路 + 状态更新 | 创建 Job → 执行 → 全部 Action 标记 Success |
| **D5** | Step 7: 分布式锁 + 心跳 | `internal/server/election/` + Agent HeartBeat | 2 个 Server 只有 1 个 Leader 工作；Agent 心跳可见 |

**里程碑**：✅ DP 端到端完整可用，可以演示完整的任务创建→执行→完成流程

## 五、优先级调整策略

### 如果只有 2 周

砍掉 Phase 3 的 P1/P2 优化和 Phase 4，只做：

- Phase 1 全部（D1-5）
- Phase 2 全部（D6-12）
- Kafka 事件驱动 + Prometheus（D13 + D17）

**结果**：核心系统 + 四大模块 + 关键优化，面试够用。

### 如果只有 1 周

只做 Phase 1 + 事务幂等性 + Saga：

- D1-5: 端到端跑通
- D6: 事务幂等性
- D7: Saga 基本框架（省略崩溃恢复的完整实现）

**结果**：系统可用 + 两个最高价值模块，面试能讲但深度有限。

## 六、风险点 & 应对

| 风险 | 概率 | 影响 | 应对 |
|------|:----:|:----:|------|
| gRPC + Proto 搭建耗时超预期 | 中 | 拖慢 Phase 1 | 参考 DTP 的 proto 定义，直接复用模式 |
| Saga 引擎复杂度超预期 | 中 | Phase 2 拖期 | 先实现简化版（无崩溃恢复），后续补全 |
| Kafka 本地环境配置问题 | 低 | 拖慢 Phase 3 | 用 Docker Compose 一键拉起 |
| AI 提效工具需要 LLM API | 低 | Phase 4 受阻 | 先做 mock 版本，面试时说"本地用 mock，线上接真实 API" |

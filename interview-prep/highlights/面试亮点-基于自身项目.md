## 阅读指南

**频率等级**（面试中被问到的概率）：
- 🔴🔴🔴 **极高频**：几乎每轮必问，不准备就挂

- 🔴 **中频**：1/3 的面试会涉及

**面试官兴趣等级**（答出来后面试官会追问的深度）：
- ⭐⭐⭐⭐⭐ **极度感兴趣**：会追问 5-10 分钟，能拉开差距

## 一、分布式事务与一致性

### 1.1 本地事务 + Outbox 模式（Transactional Outbox）

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴🔴 极高频 |
| 兴趣等级 | ⭐⭐⭐⭐⭐ |
| 典型追问 | Outbox 表怎么设计？Relay 怎么防多实例重复消费？消息顺序怎么保证？ |

**核心要点**：业务操作和消息写入在同一个数据库事务中完成，由后台 Relay 异步投递消息到 MQ。解决 DB + MQ 双写一致性问题。

**项目落点**：distributed_task_platform 的终态处理（状态更新 + 锁释放 + MQ 通知）；data-platform 的 Stage 完成事件投递。

**关键细节**：`SELECT ... FOR UPDATE SKIP LOCKED` 防多实例争抢；Outbox 表带 `(status, created_at)` 索引；TTL 清理已发送消息。

### 1.2 Saga 模式（编排式 / 协调式）

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴 高频 |
| 兴趣等级 | ⭐⭐⭐⭐⭐ |
| 典型追问 | 前向动作和反向补偿怎么设计？补偿失败了怎么办？和 2PC 的区别？ |

**核心要点**：长事务拆成多个本地事务，每个步骤有对应的补偿操作。失败时按逆序执行补偿。

**项目落点**：data-platform 的多波次灰度发布（前向：逐波次下发配置；反向：逐波次回滚到上一版本）。

**关键细节**：补偿操作必须幂等；部分补偿失败需要人工介入；Saga 状态机持久化。

### 1.3 2PC / 3PC / TCC

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴 高频 |
| 兴趣等级 | ⭐⭐⭐⭐ |
| 典型追问 | 为什么不用 2PC？2PC 的阻塞问题怎么理解？TCC 的 Try/Confirm/Cancel 各做什么？ |

**核心要点**：重点是讲清**为什么不用**——2PC 强一致但阻塞、协调者单点、性能差；TCC 业务侵入大。在微服务场景下，本地事务 + 最终一致性（Outbox/Saga）更实际。

**项目落点**：面试时作为对比方案讲，解释为什么选 Outbox 而不选 2PC/TCC。

### 1.4 最终一致性（Eventual Consistency）

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴🔴 极高频 |
| 兴趣等级 | ⭐⭐⭐⭐ |
| 典型追问 | 最终一致性"最终"是多久？怎么检测不一致？怎么修复？ |

**核心要点**：放弃强一致性，通过异步补偿、重试、对账机制保证数据最终收敛。

**项目落点**：data-platform 的 DB → Redis → Agent 三段式最终一致性；distributed_task_platform 的 Outbox + Relay 异步投递。

### 1.5 分布式事务中间件对比（Seata / DTM）

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴 中频 |
| 兴趣等级 | ⭐⭐⭐ |
| 典型追问 | 了解 Seata 的 AT 模式吗？为什么没用？ |

**核心要点**：知道有这些中间件，能说清楚为什么在自己的场景下不需要——跨服务事务场景少，核心问题是单服务内多步操作的原子性。

## 二、幂等性设计

### 2.1 接口幂等（幂等键 / Idempotency Key）

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴🔴 极高频 |
| 兴趣等级 | ⭐⭐⭐⭐⭐ |
| 典型追问 | 幂等键存哪里？过期策略？并发请求同一个幂等键怎么办？ |

**核心要点**：客户端生成唯一标识（UUID），服务端对幂等键建唯一索引，`INSERT ON CONFLICT DO NOTHING`。

**项目落点**：data-platform 的 Job 创建幂等改造（`X-Idempotency-Key` 参考 Stripe API 设计）。

### 2.2 数据库层幂等（CAS / 条件更新 / UPSERT）

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴🔴 极高频 |
| 兴趣等级 | ⭐⭐⭐⭐ |
| 典型追问 | CAS 和 SELECT FOR UPDATE 怎么选？ABA 问题怎么解？ |

**核心要点**：`UPDATE ... WHERE version = ? AND status = ?` 乐观锁；确定性 ID + 唯一索引防重复创建。

**项目落点**：distributed_task_platform 的任务抢占（CAS version）；data-platform 的 Stage 完成推进（CAS state）。

### 2.3 消费端幂等（MQ 消息去重）

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴 高频 |
| 兴趣等级 | ⭐⭐⭐⭐ |
| 典型追问 | at-least-once 和 exactly-once 区别？消费端怎么保证幂等？ |

**核心要点**：Kafka at-least-once + 消费端幂等 = 业务 exactly-once。两种方式：(1) 业务层序号条件更新 `WHERE last_sequence < new_sequence`；(2) 去重表 `(topic, partition, offset)` 唯一索引。

**项目落点**：distributed_task_platform 的执行报告消费（`last_sequence` 字段）。

### 2.4 确定性 ID 生成

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴 高频 |
| 兴趣等级 | ⭐⭐⭐ |
| 典型追问 | Snowflake 的时钟回拨怎么处理？为什么不用 UUID？ |

**核心要点**：确定性 ID（StageId-TaskIndex）天然幂等；Snowflake 全局唯一但有位溢出风险。

**项目落点**：distributed_task_platform 的 Snowflake ID（已知 Bug：序列号不绑定毫秒）。

## 三、并发控制与锁

### 3.1 分布式锁（Redis / etcd / ZooKeeper）

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴🔴 极高频 |
| 兴趣等级 | ⭐⭐⭐⭐⭐ |
| 典型追问 | Redis 主从切换锁失效怎么办？RedLock 了解吗？锁续期怎么做？ |

**核心要点**：Redis 简单互斥场景够用；etcd 需要 Watch + 选举时更适合。Lua CAS 删除防误释放。

**项目落点**：data-platform 用 Redis 做 Leader 锁（Lua CAS 释放）；distributed_task_platform 用 etcd 做服务发现和选举。

### 3.2 乐观锁 vs 悲观锁

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴🔴 极高频 |
| 兴趣等级 | ⭐⭐⭐⭐ |
| 典型追问 | 什么场景用乐观锁？冲突率高时怎么办？ |

**核心要点**：乐观锁（CAS version）适合冲突率低的场景，不阻塞；悲观锁（SELECT FOR UPDATE）适合冲突率高或必须保证互斥的场景。

**项目落点**：distributed_task_platform 全局用 CAS 乐观锁做任务抢占（冲突率低，因为 batchSize 分散）。

### 3.3 Leader 选举

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴 高频 |
| 兴趣等级 | ⭐⭐⭐⭐ |
| 典型追问 | Leader 挂了怎么切换？脑裂怎么处理？ |

**核心要点**：etcd 基于 Raft 一致性协议实现 Leader 选举；Redis 基于 TTL + Lua CAS 实现简易选举。

**项目落点**：data-platform 的 Server 端 Leader 选举（确保只有一个 Server 执行调度循环）。

### 3.4 读写锁 / 分段锁 / 无锁设计

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴 中频 |
| 兴趣等级 | ⭐⭐⭐ |
| 典型追问 | Go 的 sync.RWMutex 底层怎么实现？atomic 包怎么用？ |

**核心要点**：读多写少用 RWMutex；高性能场景用 atomic.Value（如配置热加载）；ConcurrentMap 用分段锁。

**项目落点**：data-platform 的配置热加载用 `atomic.Value` 原子替换。

## 六、流量治理与容错

### 6.1 灰度发布（金丝雀 / 蓝绿 / 滚动）

| 维度 | 评估 |
|------|------|
| 面试频率 | 🔴🔴🔴 极高频 |
| 兴趣等级 | ⭐⭐⭐⭐⭐ |

## 最后：你的项目里最能打的 10 个面试点

按"面试收益 × 项目匹配度 × 差异化程度"综合排名：

| 排名 | 面试点 | 所属项目 | 为什么能打 |
|------|--------|---------|-----------|
| 1 | **Transactional Outbox** | both | 几乎所有一致性问题都能用它收口，面试最通用 |
| 2 | **配置灰度发布 + 回滚** | data-platform | 业务味最强，状态机/事务/补偿/幂等一次全讲 |
| 3 | **PromQL 智能选节点** | distributed_task | 最大差异化亮点，同级候选人几乎没有 |
| 4 | **三项目熔断差异化选型** | 三个项目 | 横向对比是高级工程师标志 |
| 5 | **CAS 乐观锁抢占** | distributed_task | 极高频追问，answer ready |
| 6 | **四大补偿器 V1→V2** | distributed_task | 体现系统演进思维 |
| 7 | **DB/Redis/Agent 三段式一致性** | data-platform | 三段最终一致性比简单 Cache-Aside 高一个层次 |
| 8 | **三层限流 + 故障转移** | data-platform / notification | 体现限流不只是一个中间件 |
| 9 | **分片聚合 + 局部重试** | distributed_task | 体现调度内核成熟度 |
